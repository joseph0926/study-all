# loop-code

입력: `$loop-code [project-path] <goal>`

자연어 목표를 바로 코딩으로 밀어붙이지 말고, 요구사항과 계획을 먼저 합의한 뒤 섹션 단위로 구현하고 검증한다. 핵심은 "질문으로 모호함을 줄이고, 합의된 계획만 구현한다"이다.

## 모드 판별

1. `$ARGUMENTS`의 첫 토큰이 실제 경로이거나 `./`, `../`, `/`로 시작하면 **기존 프로젝트 모드**로 간주한다.
2. 기존 프로젝트 모드에서는 `<project-path> <goal>`로 파싱한다.
3. 그렇지 않으면 **새 프로젝트 모드**로 간주하고 전체를 `<goal>`로 취급한다.
4. 새 프로젝트 모드에서는 코딩 전에 프로젝트 생성 경로와 기술 스택을 먼저 확정한다.

## 세션 파일

작업 산출물은 `{project}/.codex/loop-code/` 아래에 유지한다.

- `state.md`: 현재 phase, 목표, 완료 섹션 수
- `requirements.md`: Phase 1에서 합의한 요구사항
- `plan.md`: Phase 2에서 합의한 실행 계획

첫 실행이면 `.gitignore`에 `.codex/loop-code/` 추가를 제안한다.

`state.md` 예시:

```md
# LOOP-CODE SESSION
updated: {YYYY-MM-DD HH:MM}
project: {project-path}
mode: {new|existing}
goal: {목표 요약}
phase: {N}
sections: {총 섹션 수}
completed: {완료 섹션 수}
current_section: {현재 섹션명}
```

복원 규칙:

1. Phase 0 진입 시 `state.md`를 읽는다.
2. 파일이 없거나 `# COMPLETED`이면 새 세션으로 시작한다.
3. 활성 세션이 있으면 이전 목표와 phase를 보여주고 이어서 할지 확인한다.

## 배너

실질적인 진행 응답의 첫 줄에 아래 형식을 사용한다.

```text
> [LOOP-CODE] Phase {N}/4 | {목표 요약} | 섹션 {completed}/{total}
```

## Phase 0: 초기화

### 0-A. 세션 복원 확인

1. `state.md` 존재 여부를 확인한다.
2. 활성 세션이 있으면 이어서/새로 시작 중 하나를 사용자에게 확인한다.
3. 새로 시작하면 현재 목표 기준으로 `state.md`를 초기화한다.

### 0-B. 프로젝트 환경 파악

기존 프로젝트 모드:

1. `package.json`, `pyproject.toml`, `Cargo.toml`, `go.mod` 등 대표 매니페스트를 먼저 읽어 스택을 파악한다.
2. `rg --files`와 핵심 진입점 탐색으로 디렉토리 구조를 훑는다.
3. 관련 파일을 읽어 네이밍, 폴더 구조, 테스트 패턴, 포매팅 규칙을 기록한다.
4. 기존 컨벤션을 따를지 확인하고, 특별 요청이 없으면 기존 패턴을 기본값으로 채택한다.

새 프로젝트 모드:

1. 아래 네 가지를 먼저 확정한다.
   - 프로젝트 생성 경로
   - 기술 스택
   - 패키지 매니저/런타임
   - 최소 산출물 범위
2. 사용자가 "알아서"라고 하면 합리적 기본값을 제안하고 승인받는다.

### 0-C. 초기화 보고

아래 항목을 짧게 정리하고 Phase 1로 넘어간다.

- 모드
- 경로
- 기술 스택
- 따를 컨벤션
- 현재 목표 요약

## Phase 1: 의도 구체화

추상적 목표를 검증 가능한 요구사항으로 바꾼다.

### 1-A. 질문 원칙

1. 한 번에 2~3개만 질문한다.
2. 아래 관점을 우선순위로 사용한다.
   - 범위: 포함/제외
   - 사용자와 입출력
   - 핵심 기능
   - 제약 조건
   - 성공 기준
3. 기존 프로젝트 모드에서는 통합 지점과 영향 범위를 추가로 확인한다.

### 1-B. 기본값 제안

사용자가 "모르겠다", "알아서", "아무거나"라고 답하면:

1. 합리적 기본값을 이유와 함께 제안한다.
2. 사용자가 수락하면 채택하고 다음 질문으로 넘어간다.
3. 2회 연속으로 결정이 비어 있으면 남은 항목의 기본값 묶음을 제안하고 수정 포인트만 받는다.

### 1-C. 요구사항 정리

충분히 구체화되면 아래 구조로 정리한다.

```md
## 요구사항 정리

### 목표
{한 줄 요약}

### 핵심 기능
1. {기능 1}
2. {기능 2}

### 제약 조건
- {제약 1}

### 성공 기준
- {기준}: {검증 방법}

### 제외 범위
- {제외 항목}
```

사용자가 승인하면 `requirements.md`에 기록하고 Phase 2로 진행한다.

## Phase 2: 계획 수립

합의된 요구사항을 섹션 단위 실행 계획으로 바꾼다. 이 phase에서는 코드를 작성하지 않는다.

### 2-A. 규모 판단

작업을 소/중/대로 분류한다.

- 소: 파일 1~3개, 기능 1~2개
- 중: 파일 4~8개, 기능 3~5개
- 대: 파일 9개 이상, 기능 6개 이상

분류 결과를 사용자에게 공유하고, 필요하면 섹션 수를 조정한다.

### 2-B. 전체 구조 설계

먼저 아래 두 가지를 제안한다.

1. 생성/수정할 파일 구조
2. 섹션 분할표

섹션 분할표에는 최소한 다음을 포함한다.

- 섹션명
- 책임
- 선행 의존성
- 예상 변경 파일

### 2-C. 섹션별 상세 계획

각 섹션마다 아래 네 가지를 적는다.

1. 구현 대상 파일과 역할
2. 주요 함수/컴포넌트/타입 또는 변경 포인트
3. 실행 가능한 검증 항목
4. 위험 요소와 엣지 케이스

검증 항목은 반드시 실행 가능해야 한다.

- 테스트 명령
- 타입 체크
- 린트
- 빌드
- 수동 확인 절차

### 2-D. 계획 승인

전체 계획을 요약한 뒤 `>>진행` 신호를 기다린다.

- `>>진행`: 계획 확정, `plan.md` 저장 후 Phase 3 시작
- `>>다시`: 현재 섹션 또는 전체 계획을 다시 수립

`>>진행` 없이 코딩을 시작하지 않는다.

## Phase 3: 구현

계획을 섹션 단위로 실행한다.

### 3-A. 섹션 시작

각 섹션 시작 시 아래를 짧게 보여준다.

- 현재 섹션명
- 해당 섹션의 구현 계획 요약
- 이번에 건드릴 파일
- 실행할 검증 항목

### 3-B. 코드 작성

1. 수정 전 관련 파일을 반드시 먼저 읽는다.
2. 기존 프로젝트에서는 기존 패턴과 네이밍을 따른다.
3. 계획에 없는 변경이 필요해지면 먼저 계획 수정부터 제안한다.
4. 최소 diff 원칙을 지킨다. 주변 정리나 불필요한 추상화는 추가하지 않는다.

### 3-C. 섹션 검증

섹션 완료 후 계획에 적은 검증을 실행한다.

1. 테스트
2. 타입 체크
3. 린트
4. 빌드 또는 수동 검증

실패 처리 규칙:

1. 원인을 분석하고 수정 후 재검증한다.
2. 최대 3회까지 시도한다.
3. 같은 오류가 2회 연속 반복되면 멈추고 상황을 사용자에게 보고한다.

### 3-D. 상태 갱신

섹션이 통과하면 `state.md`의 `completed`와 `current_section`을 갱신한다.

사용자가 `>>정리` 또는 `>>끝`을 입력하면 현재 안전한 단위를 마무리한 뒤 Phase 4로 이동한다.

## Phase 4: 최종 검증과 마감

### 4-A. 통합 검증

가능한 범위에서 프로젝트 수준 명령을 실행한다.

1. 전체 테스트
2. 전체 타입 체크
3. 전체 린트
4. 전체 빌드

명령이 없거나 실행 불가하면 이유와 대체 검증을 함께 보고한다.

### 4-B. 완료 보고

아래만 간결하게 정리한다.

1. 목표 대비 완료 범위
2. 섹션별 핵심 변경
3. 검증 결과
4. 남은 리스크 또는 후속 작업

### 4-C. 종료 처리

1. `state.md`를 `# COMPLETED`로 마감한다.
2. `requirements.md`와 `plan.md`는 그대로 보존한다.

## 신호 규칙

- `>>진행`: 계획 확정 후 구현 시작
- `>>다시`: 현재 계획을 다시 수립
- `>>정리` 또는 `>>끝`: 현재 안전한 단위를 마치고 최종 검증으로 이동

일반 대화 속 "진행", "정리", "다시"는 신호로 간주하지 않는다. `>>` 접두사가 있어야 한다.

## 규칙

1. Phase를 건너뛰지 않는다. 1 → 2 → 3 → 4 순서를 지킨다.
2. 계획 승인 전에는 코딩하지 않는다.
3. 검증이 통과하지 않으면 다음 섹션으로 넘어가지 않는다.
4. 한 번에 질문은 2~3개를 넘기지 않는다.
5. 사용자가 결정을 못 하면 기본값을 제안하되 거부권을 보장한다.
6. 기존 프로젝트에서는 기존 컨벤션을 우선한다.
7. 계획에 없는 큰 변경은 즉시 추가하지 말고 계획을 먼저 갱신한다.
