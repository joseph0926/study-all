## 역할 경계

- 이 스킬은 **설계가 필요한 규모의 구현 작업**에 사용한다.
- 단순 버그 수정, 한 줄 변경, 리팩터링 → 직접 처리 (이 스킬 불필요).
- 학습 목적의 코딩 → `routine` 또는 `test` 스킬 사용.
- 프로젝트 분석/개선 → `project` 스킬 사용.
- 학습 내용 적용 → `apply` 스킬 사용.

## 모드 판별

`$ARGUMENTS`의 첫 토큰이 `/`, `./`, `../`, `~/`로 시작하거나 `/`를 포함하는 상대경로 형태이고 실제 경로가 존재하면 → **기존 프로젝트**, 아니면 → **새 프로젝트**.

- 기존 프로젝트: `<project-path>` + `<목표>` 파싱
- 새 프로젝트: 전체를 `<목표>`로 취급

## 상태 보존

### 상태 파일: `{project}/.study/.loop-code/state.md`

Phase 전환마다 Write로 갱신한다:

```
# LOOP-CODE SESSION
updated: {YYYY-MM-DD HH:MM}
project: {project-path}
mode: {new|existing}
goal: {목표 요약}
phase: {N}
sections: {총 섹션 수}
completed: {완료 섹션 수}
current_section: {현재 섹션명}
```

복원:
- Phase 0 진입 시 Read (`.study/.loop-code/state.md` 우선, 없으면 `.project/state.md`도 체크하여 마이그레이션 안내)
- 파일 없음 또는 `# COMPLETED` → 새 세션
- 활성 내용 → "이전 세션: {goal}, Phase {N}, 섹션 {completed}/{sections}. 이어서 할까요?" 확인

### 배너

매 응답 첫 줄:
```
> [LOOP-CODE] Phase {N}/4 | {목표 요약} | 섹션 {completed}/{total}
```

## Phase 0: 초기화 (프로젝트 환경 파악)

### 0-A. 세션 복원 체크

1. `.study/.loop-code/state.md` 존재 확인 → 있으면 복원 제안
2. 없으면 새 세션

### 0-B. 프로젝트 환경 파악

**기존 프로젝트:**
1. `context.resolve(mode=project, projectPath=<project-path>)` 호출 → 기술 스택, CLAUDE.md, 프로젝트 구조 자동 로딩
2. `package.json` (또는 `pyproject.toml`, `Cargo.toml` 등) Read → 의존성/스크립트 파악
3. 디렉토리 구조 탐색 (Glob) → 기존 패턴(네이밍, 구조, 스타일) 분석
4. 기존 코드 컨벤션을 자동으로 파악하여 기록 (별도 질문 없이 분석 결과를 보여준다)

**새 프로젝트:**
1. 사용자에게 기술 스택 질문: "어떤 기술 스택을 사용할까요? (기본: TypeScript + React)"
2. 프로젝트 경로 확인: "프로젝트를 어디에 만들까요?"
3. `.study/.loop-code/` 디렉토리 생성 준비

### 0-C. 초기화 보고

```
## 프로젝트 환경
- 모드: {새 프로젝트|기존 프로젝트}
- 경로: {path}
- 스택: {기술 스택}
- 컨벤션: {파악된 컨벤션 요약}

목표를 구체화하겠습니다. Phase 1으로 진행합니다.
```

→ Phase 1 진행

## Phase 1: 의도 구체화 (피드백 루프)

사용자의 추상적 목표를 구체적 요구사항으로 변환한다. AI가 질문하고, 사용자 답변을 바탕으로 이해를 정제한다.

### 1-A. 핵심 질문

목표를 이해하기 위해 아래 관점에서 질문한다. 한 번에 2~3개씩 질문하여 사용자 부담을 줄인다.

1. **범위**: 무엇을 포함하고, 무엇을 제외하는가?
2. **사용자/입출력**: 누가 쓰는가? 입력은? 출력은?
3. **핵심 기능**: 반드시 있어야 하는 기능은?
4. **제약 조건**: 성능, 보안, 호환성 등 제약이 있는가?
5. **성공 기준**: 어떻게 되면 "완성"인가?

기존 프로젝트의 경우 추가로:
- 기존 코드와의 통합 지점은 어디인가?
- 기존 기능에 영향을 주는가?

### 1-B. 반복 정제

- 사용자 답변마다 이해한 내용을 요약하여 확인받는다.
- 모호한 부분이 있으면 추가 질문한다.
- 소 규모(파일 1~3개)는 1회 Q&A로 충분할 수 있다. 중/대 규모는 2회 이상.

### 1-B′. "몰라" 폴백

사용자가 "몰라", "알아서 해줘", "아무거나" 등 답변을 회피하면:

1. AI가 합리적 기본값을 **제안**한다: "그러면 X로 진행할까요? (이유: ~)"
2. 사용자가 "좋아/응" → 해당 값 채택, 다음 질문으로.
3. 사용자가 거부 → 대안 제시.
4. 1회 "몰라" → 남은 항목을 기본값으로 일괄 제안:
   "남은 결정 사항을 기본값으로 정리했습니다. 수정할 부분만 말씀해주세요."

핵심: 사용자가 모든 답을 알 필요 없다. AI가 합리적 기본값을 제안하되, 사용자에게 거부권을 보장한다.

### 1-C. 요구사항 정리

충분히 구체화되면 요구사항을 정리하여 사용자에게 보여준다:

```
## 요구사항 정리

### 목표
{한 줄 요약}

### 핵심 기능
1. {기능 1}
2. {기능 2}
...

### 제약 조건
- {제약 1}
- {제약 2}

### 성공 기준
- {기준 1}: {검증 방법}
- {기준 2}: {검증 방법}

### 제외 범위
- {제외 1}

이 요구사항이 맞나요? 수정할 부분이 있으면 말씀해주세요.
```

사용자가 승인하면 `{project}/.study/.loop-code/requirements.md`에 Write → Phase 2 진행.

## Phase 2: 계획 수립 (섹션별 피드백 루프)

요구사항을 실행 가능한 계획으로 변환한다. 섹션별로 계획하고 피드백을 받는다.

### 2-PRE. 규모 감지

계획 깊이를 작업 규모에 맞춘다. 요구사항의 핵심 기능 수와 예상 파일 수로 판단:

| 규모 | 기준 | 계획 깊이 |
|------|------|----------|
| **소** | 파일 1~3개 변경, 기능 1~2개 | 섹션 1~2개, 시그니처 생략 가능, 검증 2~3개 |
| **중** | 파일 4~8개, 기능 3~5개 | 섹션 2~4개, 주요 시그니처 포함, 검증 섹션당 3~5개 |
| **대** | 파일 9개+, 기능 6개+ | 섹션 4개+, 전체 시그니처, 검증 섹션당 4~6개 |

규모를 사용자에게 보고: "이 작업은 **{소/중/대}** 규모로 판단됩니다. 섹션 {N}개로 나누겠습니다."
사용자가 과하다고 느끼면 조절한다.

### 2-A. 전체 구조 설계

먼저 높은 수준의 구조를 제시한다:

```
## 전체 구조

### 파일 구조
{생성/수정할 파일 트리}

### 섹션 분할
| # | 섹션 | 책임 | 의존성 |
|---|------|------|--------|
| 1 | {섹션명} | {이 섹션이 하는 일} | 없음 |
| 2 | {섹션명} | {이 섹션이 하는 일} | 섹션 1 |
...

이 구조가 적절한가요?
```

사용자 피드백 반영하여 수정. 합의되면 다음.

### 2-B. 섹션별 상세 계획

각 섹션에 대해 순서대로 상세 계획을 수립한다:

```
## 섹션 {N}: {섹션명}

### 구현 내용
- {파일}: {역할}
  - {함수/컴포넌트}: {시그니처} — {설명}
  - {함수/컴포넌트}: {시그니처} — {설명}

### 테스트 계획
- {테스트 파일}: {역할}
  - {테스트 케이스}: {통과 기준}

### 검증 계획
- [ ] {테스트/검증 항목 1}: {통과 기준}
- [ ] {테스트/검증 항목 2}: {통과 기준}

### 주의사항
- {엣지 케이스, 위험 요소}

이 섹션 계획이 맞나요?
```

- 각 섹션마다 사용자 피드백을 받고 수정한다.
- **테스트 계획**: 테스트 코드의 골격(파일명, 테스트 케이스명, 검증 대상)을 이 단계에서 설계한다. 실제 코드보다 테스트를 먼저 계획하면 검증 루프가 강화된다.
- 검증 계획은 구체적이어야 한다: 테스트 코드, CLI 출력 확인, 브라우저 동작 등 **실행 가능한** 검증.
- 모든 섹션의 상세 계획이 완료될 때까지 반복한다.

### 2-C. 종합 리뷰

모든 섹션 계획이 완료되면 전체를 한눈에 보여준다:

```
## 계획 종합

### 전체 흐름
{섹션 간 의존성/실행 순서 다이어그램}

### 섹션 요약
| # | 섹션 | 파일 수 | 검증 항목 | 상태 |
|---|------|---------|----------|------|
| 1 | ... | N개 | N개 | 계획 완료 |
...

### 예상 변경 규모
- 새 파일: {N}개
- 수정 파일: {N}개
- 테스트: {N}개

>>다음 을 입력하면 코딩을 시작합니다.
수정이 필요하면 말씀해주세요.
```

사용자가 `>>다음`을 입력할 때까지 수정 피드백을 반복한다.

`>>다음` 시: `{project}/.study/.loop-code/plan.md`에 전체 계획 Write → Phase 3 진행.

## Phase 3: 구현 (섹션별 코딩 + 검증)

계획대로 섹션 단위로 코드를 작성하고 검증한다.

### 3-A. 섹션 진입

각 섹션 시작 시:

```
> [LOOP-CODE] Phase 3/4 | {목표} | 섹션 {N}/{total}: {섹션명}

### 구현 계획 (from plan.md)
{해당 섹션의 상세 계획 요약}

코딩을 시작합니다.
```

### 3-B. 코드 작성

1. **테스트 먼저**: 계획의 테스트 골격이 있으면 테스트 코드를 먼저 작성한다 (실패하는 상태로).
2. 계획의 함수/컴포넌트 단위로 구현 코드를 작성한다 (Write/Edit).
3. 기존 프로젝트의 경우 기존 코드를 먼저 Read하고 컨벤션을 따른다.
4. 한 섹션 내에서도 논리적 단위로 나눠 작성하며, 각 단위 완료 시 사용자에게 진행 상황을 보고한다.

### 3-C. 섹션 검증

섹션 코딩 완료 후, 계획의 검증 항목을 실행한다:

1. 테스트 실행 (Bash): `npm test`, `pnpm test` 등
2. 타입 체크: `tsc --noEmit` 등
3. 린트: `eslint` 등
4. 계획에 명시된 수동 검증 항목 확인

```
### 섹션 {N} 검증 결과

| # | 검증 항목 | 결과 | 비고 |
|---|----------|------|------|
| 1 | {항목} | PASS/FAIL | {상세} |
...
```

- **모두 PASS** → 섹션 완료 (3-D)
- **FAIL 있음** → 원인 분석 + 수정 → 재검증 반복.
  - 3회 실패 시: 현재 접근의 문제를 분석하고 **대안 접근을 제안**한다.
    "이 접근이 막혔습니다. 대안: (1) {대안 A}, (2) {대안 B}. 어떻게 하시겠어요?"
    사용자 선택에 따라 섹션 계획을 수정하고 재시도한다.

### 3-D. 섹션 완료

```
섹션 {N} 완료. 검증 {통과/N}개 모두 PASS.
커밋을 만들까요? (y/n)
```

1. `state.md` 갱신 (completed +1, current_section 변경).
2. **커밋 제안**: 사용자가 원하면 해당 섹션의 변경 파일을 커밋한다. 커밋은 롤백 가능한 세이브 포인트 역할을 한다.
3. 다음 섹션으로 진행.

모든 섹션 완료 시 → Phase 4 진행.

---

## Phase 4: 최종 검증 + 정리

### 4-A. 통합 검증

모든 섹션이 완료된 후 전체를 검증한다:

1. 전체 테스트 스위트 실행
2. 타입 체크 (전체)
3. 린트 (전체)
4. 빌드 확인 (해당되는 경우)

```
### 통합 검증 결과

| # | 항목 | 결과 |
|---|------|------|
| 1 | 테스트 | PASS (N/N) |
| 2 | 타입 체크 | PASS |
| 3 | 린트 | PASS |
| 4 | 빌드 | PASS |
```

실패 시 → 원인 분석 + 수정 → 재검증.

### 4-B. 완료 보고

```
## 구현 완료

### 목표
{요구사항에서 정의한 목표}

### 변경 요약
| # | 섹션 | 파일 | 변경 |
|---|------|------|------|
| 1 | {섹션명} | {파일 목록} | +{N}, -{M} |
...

### 검증 결과
- 섹션별 검증: 모두 PASS
- 통합 검증: PASS

### 생성된 파일
{새로 생성된 파일 목록}

### 후속 작업 (있다면)
- {추가로 할 수 있는 개선 사항}
```

### 4-C. 정리

1. `state.md` → `# COMPLETED` 갱신
2. `.study/.loop-code/plan.md`, `.study/.loop-code/requirements.md`는 보존 (이력 참조용)
3. `session.appendLog(context, topic=<목표 요약>, content=<완료 보고 요약>, via="via /loop-code")` — 세션 이력을 대시보드에 반영

- 사용자가 계속을 원하면 진행, 아니면 현재 Phase 완료 후 Phase 4로 유도.

## 사용자 신호 규칙

- `>>다음` — Phase 2 계획 확정 → Phase 3 코딩 시작
- `>>정리` 또는 `>>끝` — 현재 진행 중인 작업 마무리 후 Phase 4 실행
- `>>다시` — 현재 섹션 계획을 처음부터 다시 수립
- 일반 대화 속 "다음", "다시" 등은 신호로 인식하지 않는다 (`>>` 접두사 필수).

---

## 규칙

- AI가 Phase를 건너뛰지 않는다. 0→1→2→3→4 순서를 따른다.
- Phase 2에서 `>>다음` 없이 코딩을 시작하지 않는다.
- 검증이 통과하지 않으면 다음 섹션으로 넘어가지 않는다.
- 코드 작성 시 기존 프로젝트 컨벤션을 따른다.
- 계획에 없는 코드를 작성하지 않는다. 추가가 필요하면 사용자에게 계획 수정을 제안한다.
- `.study/` 디렉토리를 `.gitignore`에 추가 안내 (첫 실행 시).
- 한 번에 질문은 2~3개를 넘지 않는다. 사용자 부담을 줄인다.
- `session.appendLog` 호출 실패 시 경고만 출력하고 진행한다 (MCP 미사용 환경 대응).
