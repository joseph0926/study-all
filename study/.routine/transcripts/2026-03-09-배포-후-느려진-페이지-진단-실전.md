좋습니다. 매핑은 정확합니다. 조금 더 정밀하게 잡아보겠습니다.

### LCP (Largest Contentful Paint)

"페이지가 렌더링되는데 걸리는 시간"은 맞지만, 정확히는 **뷰포트 안에서 가장 큰 콘텐츠 요소**(이미지, 텍스트 블록 등)가 화면에 그려지는 시점입니다.

- **비유**: 카페에서 주문 후 메인 요리가 나오는 시간. 물이나 냅킨(FCP)은 먼저 나와도, 손님이 "왔다"고 느끼는 건 메인 요리(LCP)가 도착할 때.
- 전체 렌더링 완료가 아니라 **"사용자가 핵심 콘텐츠를 봤다"는 인지 시점**을 측정하는 것.
- 임계값: **2.5초** 이내 good, 4초 초과 poor.

### INP (Interaction to Next Paint)

"인터랙션 후 페인트까지 걸리는 시간" — 정확합니다. 단, 포인트가 하나 있습니다:

- INP는 단일 인터랙션이 아니라 **세션 전체에서 가장 느린 인터랙션**(상위 근처)을 대표값으로 잡습니다.
- **비유**: 식당 평가에서 "가장 오래 기다린 주문"이 그 식당의 응답성 점수가 되는 것.
- 임계값: **200ms** 이내 good, 500ms 초과 poor.

### CLS (Cumulative Layout Shift)

"화면이 움직이거나 어긋나는 수치" — 맞습니다.

- 사용자 입력 없이 발생한 레이아웃 이동의 **누적 점수**.
- 임계값: **0.1** 이내 good, 0.25 초과 poor.

암기보다는 **흐름을 이해하면 자연스럽게 나옵니다.** 브라우저가 화면을 그리려면 거치는 단계를 생각해보세요.

### LCP가 느린 4가지 하위 단계

비유: 온라인 쇼핑몰에서 상품 페이지를 여는 과정

| 단계 | 비유 | 실제 | 도구 |
|---|---|---|---|
| **1. TTFB** | 택배 회사가 물건을 보내기 시작하는 시간 | 서버가 첫 바이트를 응답하기까지 | Network 탭 (Waiting 시간) |
| **2. Resource load delay** | 물건이 도착했지만 포장 안에 "이 부품도 필요"라는 메모가 있어서 추가 주문 | HTML 도착 후 LCP 리소스(이미지 등) 요청이 시작되기까지의 지연 | Network waterfall |
| **3. Resource load time** | 추가 부품이 배송되는 시간 | LCP 리소스(이미지, 폰트 등)가 다운로드되는 시간 | Network 탭 (Content Download) |
| **4. Render delay** | 부품이 다 왔는데 조립하느라 기다리는 시간 | 리소스 도착 후 실제 화면에 그려지기까지 | Performance 탭 (Main thread) |

```
TTFB → Resource load delay → Resource load time → Render delay
 ↓           ↓                    ↓                   ↓
서버 느림?   CSS/JS가 막고 있나?   이미지가 큰가?      JS가 메인스레드 점유?
```

### 왜 암기가 아닌가

"브라우저가 HTML을 받고 → 필요한 리소스를 발견하고 → 다운로드하고 → 그린다" — 이 4단계는 브라우저가 동작하는 순서 그 자체입니다. 어디서 병목이 걸렸는지를 **Network waterfall**과 **Performance 패널**에서 구간별로 보면 됩니다.

### Resource load delay가 맞을까?

절반만 맞습니다. 상황에 따라 다릅니다:

| 시나리오 | 의심 단계 | 이유 |

| API가 느려서 렌더링 자체를 블로킹하는 경우 (예: React에서 fetch 완료 후 setState → 리렌더) | **Render delay** | 리소스는 있지만 JS가 메인스레드를 잡고 있어서 페인트가 늦음 |
| API가 서버 사이드에서 호출되는 경우 (SSR) | **TTFB** | 서버가 API 응답 기다리느라 HTML 자체를 늦게 보냄 |

**도구는 맞습니다** — Network 탭에서 waterfall을 보면 새 API 호출이 어디에 끼어들어서 LCP 리소스 로드를 지연시키는지 바로 보입니다.

맞습니다. 메인 스레드 점유가 핵심 원인입니다. 좀 더 구체적으로 분해해보겠습니다.

### INP가 높은 3가지 원인

INP = **Input delay** + **Processing time** + **Presentation delay**

```
클릭 발생 ──→ 이벤트 핸들러 시작 ──→ 핸들러 완료 ──→ 다음 페인트
         ↑                      ↑                 ↑
    Input delay           Processing time    Presentation delay
   (대기 시간)             (실행 시간)         (렌더링 시간)
```

| 구간 | 비유 (식당) | 원인 예시 | DevTools 확인 |
|---|---|---|---|
| **Input delay** | 주문했는데 직원이 다른 테이블 서빙 중 | 클릭 시점에 이미 long task가 메인스레드 점유 중 | Performance 탭 → 클릭 이벤트 앞의 long task |
| **Processing time** | 직원이 주문을 받았는데 조리가 오래 걸림 | 이벤트 핸들러 자체가 무거움 (거대 setState, 동기 계산) | Performance 탭 → 이벤트 핸들러 call tree |
| **Presentation delay** | 요리는 됐는데 플레이팅이 오래 걸림 | 핸들러 후 리렌더/레이아웃/페인트가 무거움 | Performance 탭 → 핸들러 이후 렌더링 구간 |

세 구간 모두 **Performance 탭**에서 확인합니다. 클릭 이벤트를 녹화하면 해당 인터랙션이 표시되고, 어떤 구간이 긴지 바로 보입니다.

**면접 답변 프레임**: *"INP가 높다면 Performance 탭에서 해당 인터랙션을 녹화합니다. Input delay가 크면 기존 long task를 분할하고, Processing time이 크면 핸들러 로직을 경량화하고, Presentation delay가 크면 리렌더 범위를 줄입니다."*

## Phase 2: 심화 — "왜 이렇게 만들었는가"

Phase 1에서 **증상→지표→하위 단계→도구** 프레임을 잡았습니다. Phase 2에서는 이 프레임이 **왜 이런 구조인지**, 그리고 **실전에서 어디서 판단이 흔들리는지**를 파고듭니다.

### 설계 의도 추적

Web Vitals가 왜 LCP/INP/CLS 이 3개로 수렴했는지 생각해봅시다.

과거에는 성능 지표가 **기술 중심**이었습니다:
- `DOMContentLoaded`, `load` — 브라우저 내부 이벤트
- `First Paint`, `First Contentful Paint` — 픽셀 기준

문제: 이 지표들이 좋아도 **사용자는 "느리다"고 느낄 수 있었습니다.** `load` 이벤트가 빨라도 메인 콘텐츠가 안 보이면 느린 거고, FCP가 빨라도 보이는 건 로딩 스피너뿐이면 의미 없습니다.

그래서 Google이 **사용자 체감 중심**으로 재설계한 것이 Core Web Vitals입니다:
- "핵심 콘텐츠가 보이는가?" → LCP (FCP 대체가 아니라 보완)
- "반응하는가?" → INP (FID를 대체 — FID는 첫 인터랙션만 측정했으나, INP는 전체 세션)
- "안정적인가?" → CLS

**트레이드오프**: 지표가 3개로 적어지면 측정이 단순해지지만, 모든 상황을 커버하지 못합니다. 예를 들어 **스크롤 버벅임(jank)** 은 LCP/INP/CLS 어디에도 직접 잡히지 않습니다. 이건 long task / 프레임 드랍으로 별도 추적해야 합니다.

    const { largestShiftTarget } = metric.attribution;
    console.log('[CLS]', rate(metric.value, 0.1, 0.25), {
      largestShiftTarget,
    });
  });
}
